# 估值表解析器项目任务书 v3

> 适用对象：基于脱敏后的估值表样本，开发“按产品-托管机构映射自动路由 + 按托管券商/托管行规则解析估值表”的标准化工具。  
> 本任务书默认：产品名称脱敏已由上游私有工具完成；开发者不接触真实产品名称，也不接触脱敏词典。  
> 新增前提：上游会额外提供一份“产品 -> 托管机构/托管券商”映射表，用于自动选择解析 adapter，并为后续生产接入预留接口。  
> 本版新增：将单项目 Git 提交规范正式纳入项目任务书，作为开发执行要求的一部分。

---

## 1. 项目背景

我们已经完成了估值表样本的脱敏处理。你将接手的是**脱敏后的估值表样本**，不接触真实产品名称，也不接触脱敏词典。

现在新增一项上游输入：**产品与托管机构映射表**。  
这张表的作用不是做业务分析，而是做**解析路由（routing）**：

- 根据脱敏产品标识或关联代码，识别该估值表应该走哪个 adapter
- 将“按产品识别”与“按托管机构规则解析”分开
- 让同一个托管机构下的多个产品，复用同一套解析规则
- 为后续真实生产系统接入保留稳定接口

因此，本项目不再只是“按券商手工指定解析”，而是升级为：

> **先识别产品身份 -> 再查映射表 -> 再选择对应托管机构 adapter -> 最后做结构化解析**

本项目重点不是做 Excel 展示，而是做**估值表标准化解析 + 自动路由骨架**。

---

## 2. 项目目标

构建一个**按产品映射自动路由，并按托管机构规则解析估值表的标准化工具**。

工具需要完成以下目标：

### 目标 A：读取估值表
支持读取脱敏后的估值表文件，优先支持：

- `.xls`
- `.xlsx`

### 目标 B：读取产品-托管机构映射表
支持读取一份额外映射表，优先支持：

- `.csv`
- `.xlsx`

映射表用于将：
- 脱敏产品编号
- 关联代码
- 托管机构标识
- adapter 标识

统一收敛为**解析路由输入**。

### 目标 C：自动路由到正确 adapter
根据文件名、sheet 标题、表头文本等提取产品身份，再查映射表，自动确定应该使用哪个 adapter。

### 目标 D：识别估值表中的科目结构
将估值表中的科目行识别出来，并保留必要的层级关系与原始痕迹。

### 目标 E：抽取持仓叶子节点
从科目结构中识别出“真实持仓”对应的叶子节点。

### 目标 F：标准化证券代码
将持仓对应的原始编码，转换为统一格式，例如：

- `600109.SH`
- `000560.SZ`
- `09926.HK`
- `511360.SH`

### 目标 G：输出标准化结果
至少输出三类结构化结果：

1. **科目明细表**
2. **持仓标准表**
3. **路由结果表**

### 目标 H：输出解析摘要
生成 summary，说明：

- 处理文件数
- 成功自动路由数
- 手动 override 数
- 识别到多少科目
- 提取到多少持仓
- 哪些记录未识别
- 哪些记录需要人工复核

## 3. 输入与输出

## 3.1 输入

本项目至少包含两类输入：

### 输入 A：脱敏后的估值表样本
- 每份样本至少已去除或替换产品名称
- 文件来源可能对应不同托管券商 / 托管行
- 文件命名模式不统一，不能只靠单一正则判断

当前样本可见的文件名模式至少包括：

- `2025-03-27_PRODUCT_001估值表.xlsx`
- `20250327_PRODUCT_002_证券投资基金估值表.xls`
- `PRODUCT_006_资产估值表_20250327.xls`
- `PRODUCT_008委托资产资产估值表20250327.xls`
- `PRODUCT_010_证券投资基金估值表_2025-03-27.xls`
- `PRODUCT_012_估值表_20250327.xls`
- `XXX007_PRODUCT_007_估值表_20250327.xls`
- `估值表_PRODUCT_021_20250327.xls`
- `估值表日报-XXX022-PRODUCT_022-4-20250327.xlsx`
- `证券投资基金估值表_PRODUCT_023_2025-03-27.xlsx`

这说明：
- 产品编号可能出现在文件名中间、开头、结尾
- 可能伴随关联代码（如 `XXX007` / `XXX022`）
- 可能伴随“证券投资基金估值表 / 资产估值表 / 估值表日报”等不同前后缀

### 输入 B：产品-托管机构映射表
这是一张额外维护的映射表，生产上可来自 Excel 或 CSV。

建议至少包含以下字段：

- `product_id`：脱敏产品编号，例如 `PRODUCT_001`
- `association_code`：关联代码，可选，例如 `XXX001`
- `custodian_id`：托管机构稳定标识，例如 `CUSTODIAN_007`
- `custodian_name`：托管机构名称，可为真实名称或开发态假名
- `adapter_key`：解析器实际使用的 adapter 标识，例如 `guosen` / `greatwall`
- `is_active`：是否启用
- `note`：备注，可选

说明：
- **`custodian_id` / `adapter_key` 才是规则归属主键**
- `product_id` 只是路由入口，不应让每个产品都派生一套解析规则

## 4. 范围边界

## 4.1 本期必须做

- 读取 `.xls/.xlsx` 估值表
- 读取 `.csv/.xlsx` 映射表
- 从文件名 / sheet / 表头中提取 `product_id` 与 `association_code`
- 根据映射表自动选择 adapter
- 保留手动 `--adapter` override 作为开发/排障兜底
- 识别科目行
- 识别持仓叶子节点
- 按托管机构规则解析
- 统一证券代码格式
- 输出标准 CSV
- 生成解析 summary
- 补最小 smoke test

## 4.2 本期不要做

- 产品名称脱敏
- 真实词典管理
- Wind / 行情补数
- 持仓穿透到底层资产
- 多日估值表串联分析
- 与交易、指令、结算数据自动对账
- 复杂报表美化
- 用大模型“猜”托管机构，不允许用 AI 猜规则代替显式映射

## 5.2 设计原则

- **路由逻辑与解析逻辑必须分离**
- `product_id -> custodian_id -> adapter_key` 必须可追踪
- 证券代码标准化必须单独成模块
- 输出字段要尽早收敛，避免每个托管机构各搞一套
- 必须保留必要的原始痕迹，方便回查和人工复核
- 先做最小闭环，再扩第二家 / 第三家托管机构

## 6. 规则设计要求

## 6.1 产品身份提取（Product Identity）
必须先做一个独立模块，负责从以下位置提取身份信息：

- 文件名
- sheet 名
- 表头标题文本
- 专用表抬头

优先提取：

- `product_id`，例如 `PRODUCT_021`
- `association_code`，例如 `XXX021`

注意：
- 样本命名不统一
- 不能假设 `product_id` 一定出现在固定位置
- 允许多个正则模式并行尝试
- 提取失败时，必须输出明确路由失败原因

## 6.2 映射表加载（Mapping Loader）
映射表必须由单独模块负责读取、校验与标准化。

至少校验：

- `product_id` 是否重复冲突
- `association_code` 是否重复冲突
- `adapter_key` 是否存在于注册表
- `is_active` 是否可正确过滤

开发态允许使用脱敏 mapping；生产态允许切换到真实 mapping，但代码不应因字段语义变化而重写。

## 6.3 路由（Routing）
推荐优先级：

1. 手动 `--adapter` override
2. `product_id + association_code` 命中映射
3. 仅 `product_id` 命中映射
4. 仅 `association_code` 命中映射
5. header 中直接识别到稳定托管机构名称（仅作为辅助，不应替代映射表）

必须输出：
- 路由成功 / 失败
- 命中依据
- 使用的 adapter

## 6.4 托管机构适配（Adapter）
每家托管机构单独实现一个 adapter，至少负责：

- 表头定位
- 列含义映射
- 科目行识别
- 叶子持仓识别
- 特定资产类型识别
- 异常记录打标

**adapter 应该以托管机构为单位，而不是以产品为单位。**

## 6.5 代码标准化
证券代码标准化必须独立成公共模块，不要写死在单个 adapter 脚本里。

至少覆盖：

- 沪市 A 股 -> `.SH`
- 深市 A 股 -> `.SZ`
- 港股 -> 5 位补零 + `.HK`
- 场内基金 / ETF -> 补交易所后缀
- 无法识别时输出 `review_flag`

## 6.6 原始痕迹保留
必须保留必要 trace，方便排错：

- 原始行号
- 原始科目代码
- 原始科目名称
- 原始证券代码文本
- 来源文件名
- 来源 sheet 名
- 提取到的 `product_id`
- 提取到的 `association_code`
- 命中的 `custodian_id`
- 命中的 `adapter_key`

## 9. Git 与提交规范

本项目使用**轻量结构化的单项目 commit 规范**。  
目标不是把提交写得更正式，而是让 Git 历史能清楚表达：

- 这次改动做了什么
- 属于新功能、修复、收敛、文档还是测试
- 是否属于某个明确阶段
- 是否改变了外部行为或运行契约

## 9.1 基本格式

推荐使用以下三种格式之一：

```text
<type>: <action>
<type>(<scope>): <action>
<type>: Phase <n>[.<m>] - <scope> - <action>
```

示例：

```text
feat(routing): add mapping-based adapter selection
fix(parser): handle empty subject rows in xls samples
refactor(adapter): isolate shared leaf-position detection
docs(handoff): refresh current boundaries and next steps
test(smoke): add routing-to-parser minimal coverage
docs: Phase 1 - routing - define mapping-driven parse flow
```

## 9.2 type（改动类型）

默认只使用以下六类：

- `feat`：新功能、新入口、新能力
- `fix`：bug 修复、错误路径修正、兼容性修补
- `refactor`：结构整理、实现收敛，不改变外部行为
- `docs`：文档、handoff、status、说明更新
- `test`：测试、smoke、回归保护
- `chore`：杂项维护、配置、小型清理

## 9.3 推荐 scope（适配当前项目）

本项目常用 scope 建议控制在少量稳定集合内：

- `routing`
- `mapping`
- `identity`
- `parser`
- `adapter`
- `normalization`
- `export`
- `config`
- `cli`
- `pipeline`
- `smoke`
- `handoff`
- `status`

scope 不强制，但当改动明显集中在某一层时，建议写上。

## 9.4 action 写法要求

action 应简短说明“本次做了什么”。

推荐写法：

- `add mapping-based adapter selection`
- `extract shared code normalization helper`
- `preserve xls workbook sheet order`
- `refresh handoff after routing smoke pass`
- `stabilize adapter registry behavior`

不推荐写法：

- `update`
- `misc`
- `fix bug`
- `do changes`
- `continue work`

## 9.5 Phase 的使用原则

Phase 是**可选项**，只在项目确实存在阶段叙事时使用。

### 可以写 Phase 的情况
- 当前提交是某阶段的边界定义、阶段收口或里程碑
- 你希望 Git log 能直接映射项目演进时间线

例如：

```text
docs: Phase 0 - routing - define product identity and mapping flow
refactor: Phase 1 - adapter - extract shared position parsing helper
test: Phase 1.1 - smoke - cover first custodian end-to-end flow
```

### 不要写 Phase 的情况
- 普通小修
- 一次性的检查
- 一轮 AI 执行步骤，如 `scan / audit / report / fix`
- 当天的随手调整

判断口径：

> **项目阶段可以写 Phase，施工步骤不要写 Phase。**

## 9.6 本项目最常见的高质量提交示例

### 路由 / 映射
- `feat(routing): add mapping-based adapter selection`
- `fix(mapping): reject duplicated active product mappings`
- `test(identity): cover product id extraction across file name variants`

### 解析 / adapter
- `feat(parser): add first custodian valuation entry`
- `refactor(adapter): extract shared leaf-position detection`
- `fix(parser): handle empty subject rows in xls samples`

### 标准化 / 导出
- `refactor(normalization): centralize security code normalization`
- `fix(export): preserve source file and route metadata in outputs`

### 文档 / 状态
- `docs(handoff): refresh routing boundaries and next steps`
- `docs(status): record phase-1 parsing snapshot`

## 9.7 commit body 的使用建议

默认优先单行 subject，不强制写 body。  
但以下情况建议加 body：

- 这次改动有明显边界约束
- 做的是“低风险收敛”，需要说明没做什么
- 兼容旧入口或旧输出
- 单行 subject 无法表达“行为不变”这类约束

推荐格式：

```text
refactor(adapter): extract shared leaf-position detection

- keep current output fields unchanged
- preserve adapter routing contract
- reduce duplicated parsing logic across two custodians
```

## 9.8 开发执行要求

- 每次开始一轮 AI 改动前，先 commit 一次
- 每个阶段结束后提交一次清晰 commit
- 不要把临时输出、缓存文件、原始敏感文件提交进仓库
- 不要把多个不相关改动压成一条 commit
- 对 migration / 收敛类改动，优先写清是否改变外部行为

## 12. 建议 CLI

### 自动路由模式（默认）
```bash
python -m valuation_parser.cli \
  --input "data_samples/raw/" \
  --mapping "data_samples/mapping/custodian_mapping.csv" \
  --output-dir "output/"
```

### 手动 override 模式（开发 / 排障）
```bash
python -m valuation_parser.cli \
  --input "data_samples/raw/sample.xls" \
  --mapping "data_samples/mapping/custodian_mapping.csv" \
  --adapter "greatwall" \
  --output-dir "output/"
```

- `--input`
- `--mapping`
- `--output-dir`
- `--summary-path`
- `--adapter`
- `--fail-on-routing-error`

## 14. 初始建议

### 14.1 推荐先支持的资产类型
优先做：

- A 股
- 港股
- 场内基金 / ETF
- 现金 / 存款 / 保证金
- 基础负债科目单独落表

### 14.2 暂缓的复杂场景
先不要求完整处理：

- 收益互换穿透到底层证券
- 复杂衍生工具估值结构
- 特殊停牌估值场景
- 多账户合并与跨表对齐

### 14.3 目标导向
第一阶段目标不是“覆盖所有托管机构和所有科目”，而是：

> 做出一个可扩展、可测试、可复用的最小解析骨架，并在自动路由 + 一到两家托管机构样本上稳定跑通。

### 14.4 生产接入导向
后续接生产时，推荐保持以下分层：

- **上游私有层**：真实产品名脱敏、真实敏感映射维护
- **解析路由层**：产品身份提取 + 映射查找 + adapter 选择
- **解析执行层**：按托管机构规则提取结构化数据
- **下游数据中心层**：统一表落库、质量校验、进一步联通估值/交易/行情数据

也就是说：

> 实习生开发的解析器项目，应尽量做成“中间标准层”，而不是把敏感处理和生产特定逻辑混在一起。
